## 发现 #1：Scrapling 技能未注册（P2-低风险）

### 问题描述
`workspace/skills/scrapling/` 技能文件存在且功能完整（自适应爬虫、Playwright 动态渲染、反爬），
但 AGENTS.md 技能注册表中未录入，导致龙虾开新会话时不知道自己有这项能力。

### 根因
技能安装时跳过了 SKILL_LIFECYCLE.md 第 5 步（技能注册），属于运维遗漏。

### 影响
- 功能层面：scrapling 可正常使用（文件完整）
- 能力感知层面：新会话龙虾不会主动使用，需主人提示

### 修复措施
- 在 AGENTS.md 技能注册表末尾添加 scrapling 条目
- AGENTS.md 版本从 v5 升级到 v6
- 更新 AGENTS.md checksum

### 经验教训
每次安装新 Skill 后必须严格执行 SKILL_LIFECYCLE.md Step 5（注册）。

## 发现 #2：AI_RULES.md 缺少认证旁路禁止条款（P1-中风险）

### 问题描述
AI_RULES.md v3 中没有明文禁止创建 `.pre-disable-auth` 文件（该文件一旦存在，
Gateway 会跳过所有认证，任何人无需 Token 即可访问龙虾）。

早期安全加固时仅通过**物理删除**该文件来解决问题，没有在规则层面进行封堵，
属于规则覆盖盲区：若龙虾日后被诱导操作，规则里没有明文禁止，可能执行。

### 根因
安全加固聚焦于物理修复（删文件），忽略了规则层补丁。

### 影响
- 若攻击者通过 Prompt Injection 诱导龙虾创建 `.pre-disable-auth`，
  可完全绕过 Gateway 认证，获得无限制访问权限
- 属于潜在 P0 风险的前置条件

### 修复措施
- 在 AI_RULES.md 中新增 L0.10 章节，明确禁止：
  1. 创建 `~/.openclaw/.pre-disable-auth`
  2. 修改 `gateway.auth` / `gateway.bind` 为不安全值
  3. 任何绕过 Tailscale HTTPS 暴露 Gateway 端口的操作
- AI_RULES.md 版本从 v3 升级到 v4

### 经验教训
物理修复（删除旁路文件）必须配合规则修复（AI 行为约束），缺一不可。

## 发现 #3：healthcheck.sh 缺失（P2-低风险 / 部分误报）

### 问题描述
龙虾自检报告称 `~/.openclaw/scripts/` 下缺少 `healthcheck.sh`。

### 根因分析
**部分属实，部分误报：**
- guardian-core.md v1.0.0 中确实未定义 healthcheck.sh，龙虾存在超出设计范围的预期
- 但从功能角度，healthcheck.sh 是 G1 启动护盾的合理延伸，缺少它意味着 G1 的完整性
  检查、端口绑定、供应链验证等都无法一键执行

### 修复措施
- 创建 `~/.openclaw/scripts/healthcheck.sh`（G1 启动健康检查脚本 v1.0）
- 涵盖 7 项检查：AGENTS.md 校验 / 认证旁路 / Gateway 进程 / 端口绑定 / 供应链 denylist / HF Token 权限 / requirements hash 锁
- 首次执行结果：5 PASS，0 WARN，0 FAIL（修复 checksum 后）

### 经验教训
安全架构文档（guardian-core.md）应在定义每个 G 层时，同步明确配套脚本的路径和名称，
避免"规范存在但工具缺失"的落地断层。

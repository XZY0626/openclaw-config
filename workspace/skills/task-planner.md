# task-planner — 任务规划技能 v2（WorkBuddy 风格）

## 用途

收到复杂任务时，先规划再执行。像 WorkBuddy 一样：**结构清晰、并行处理、透明展示进度**。

---

## 何时启动任务规划？

- 任务步骤 ≥ 3 步
- 涉及多个文件/工具/服务
- 有破坏性操作（删除/覆盖/配置修改）
- 主人给了多个并列任务

简单问答（"xxx是什么意思"/"帮我查下"）不需要规划，直接答。

---

## 规划流程

### Step 1：理解目标

一句话确认主人真正想要的结果：
```
目标：将 openclaw 工具权限从 messaging 扩展到包含 exec/read/write 的完整列表
```

### Step 2：拆解步骤

列出所有子任务，标注**依赖关系**和**并行机会**：

```markdown
任务分解：

[可并行执行]
  A. 读取当前 openclaw.json
  B. 读取 AI_RULES.md 确认合规边界

[依赖 A 的结果]
  C. 分析当前权限 vs 目标权限，列出差异

[可并行执行]
  D. 备份当前配置
  E. 准备新配置内容

[依赖 D+E]
  F. 写入新配置，重启 gateway

[依赖 F]
  G. 验证权限生效

[最后执行]
  H. 写日志 → 推 GitHub
```

### Step 3：识别风险

```markdown
风险点：
⚠️ F（写入配置）是破坏性操作 → 需要 D（备份）完成后才执行
⚠️ G（重启 gateway）会短暂中断服务 → 告知主人
```

### Step 4：向主人展示计划（可选，视任务复杂度）

超过5步或有高风险操作时，先给主人看计划：
```
我准备分三个阶段来做：
1. [读现状] 同时读取配置文件和规则文件
2. [修改] 备份 → 写新配置 → 重启验证
3. [收尾] 日志 + GitHub

有问题吗？没问题我就开始了。
```

## 并行执行规范

独立任务一定并行，节省时间：

**原则：** 如果 A 的结果不影响 B 的执行，就同时启动 A 和 B。

**标记方式：**
```
[并行启动]
  → 读取 SOUL.md
  → 读取 USER.md  
  → 读取 AI_RULES.md
（三个同时，等全部完成后继续）
```

---

## 任务规模快速判断

| 步骤数 | 策略 |
|--------|------|
| 1-2步 | 直接做，不需要规划 |
| 3-5步 | 内部规划，执行时透明展示进度 |
| 6步以上 | 先给主人看计划，确认后再执行 |
| 有破坏性操作 | 无论步数多少，必须展示计划确认 |
